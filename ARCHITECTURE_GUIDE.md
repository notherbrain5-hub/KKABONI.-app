# 🏗️ KKABONI (까보니) 코어 아키텍처 및 확장 가이드

이 문서는 앞으로 스드메, 웨딩홀, 본식스냅 등 수많은 품목과 AI 에이전트들이 추가되더라도 **코드 충돌(Conflict)과 병목현상을 방지하기 위한 구조 가이드**입니다. 
새로운 기능을 추가하거나 크롤러를 수정할 때 이 문서를 기준으로 작업해 주세요.

---

## 1. 📁 폴더 및 파일 분리 원칙 (역할 철저 분리)

모든 코드는 절대 하나의 파일에 섞이지 않으며, 아래의 구역에 맞게 배치되어야 합니다.

- `src/config/` : 환경 설정 및 DB 접속, 공통 더미 데이터
- `src/crawlers/` : 외부 데이터를 수집하고 파싱(추가금 분석)하는 크롤러 엔진
- `src/components/` : 화면(UI)을 그리고 이벤트를 관리하는 지휘소

---

## 2. 🕷️ 크롤러(Crawler) 확장 및 충돌 방지 규칙

앞으로 본식스냅, 예물, 예복 등 새로운 품목의 크롤러를 추가할 때는 기존 코드를 건드리지 않고 **새로운 파일을 생성**하여 확장합니다.

### ✅ 올바른 확장 방법 (예시: 본식스냅 추가 시)
1. `src/crawlers/` 폴더 안에 `snapCrawler.js` 라는 독립된 파일을 만듭니다.
2. `utils.js`에 있는 공통 함수(`fetchHtml`, `extractNumbers`)만 가져와서 사용합니다.
3. 스냅 전용 키워드(예: '원본 파일 필수', '2인 작가 추가금') 정규식만 `snapCrawler.js` 내부에 작성합니다.
4. `src/components/app.js` 에서는 `import { fetchSnapData } from '../crawlers/snapCrawler.js'` 처럼 필요한 것만 가져다 씁니다.

### ❌ 절대 하면 안 되는 행동 (에러 유발)
- `hallCrawler.js` 안에 스드메나 스냅 로직을 섞어 적기 (웨딩홀 구조가 바뀌면 스드메까지 고장남)
- `utils.js`에 특정 품목 전용 키워드 우겨넣기 (공통 유틸리티가 무거워짐)

---

## 3. 🤖 AI 에이전트(Agent) 파이프라인 규칙

Agent1~4는 각각 데이터를 수집, 필터링, 요약, 검증하는 독립적인 객체입니다. 에이전트 간 데이터 오염을 막기 위해 아래의 파이프라인 규칙을 준수합니다.

### [Agent 1] 데이터 수집 아키텍트 (`crawlers/`)
- **역할**: 외부 웹사이트 HTML을 가져와 추가금 키워드를 찾아내고 DB 모델(JSON) 형태로 정제합니다.
- **규칙**: UI 화면을 절대 직접 조작하지 않습니다. 오직 순수한 '데이터'만 반환(Return)하거나 Supabase에 넘깁니다.

### [Agent 2] 수집 및 정형화 (`mockAgents.js`)
- **역할**: DB에 쌓인 수만 개의 데이터를 지역별/품목별로 꺼내오는(Fetch) 필터링 역할을 합니다.

### [Agent 3] 요약 및 스코어링 (`mockAgents.js`)
- **역할**: Agent 2가 꺼내온 데이터를 바탕으로 '안심 지수(Transparency Score)'와 '숨겨진 추가금 비율'을 수학적으로 계산합니다.
- **규칙**: 원본 데이터를 덮어쓰지 않고, 연산 결과가 포함된 **새로운 객체를 반환**해야 합니다. (불변성 유지)

### [Agent 4] 영수증 교차 검증 (`mockAgents.js`)
- **역할**: 텍스트(리뷰)와 이미지(영수증)를 비교하여 광고성/거짓 리뷰를 찾아냅니다.
- **규칙**: `utils.js`의 `isAdvertisement()` 공통 함수를 활용하며, 점수에 따라 `"Verified"`, `"Rejected"` 상태코드만 명확히 반환합니다.

---

## 4. 🗄️ Supabase 데이터베이스 규칙

데이터베이스 스키마(테이블 모양)가 무분별하게 변경되는 것을 막기 위한 필수 규칙입니다.

1. **테이블 구조 변경 금지**: `id`, `name`, `base_price`, `hidden_cost`, `created_at` 등 기본 컬럼의 이름을 앱 내부에서 임의로 바꾸어선 안 됩니다. 
2. **품목 추가 시 대처법**: 웨딩홀, 스드메 등 품목 구분이 필요해질 경우, 테이블을 새로 만들기보다는 `wedding_halls` 테이블의 이름을 `vendors`로 통합하고 내부에 `category(TEXT)` 컬럼을 추가하는 방식으로 마이그레이션(Migration)하는 것이 안전합니다.
3. **직접 접근 금지**: 컴포넌트(`Dashboard.js`) 외의 크롤러 파일에서 DB에 마구잡이로 데이터를 쏴버리면 트래픽이 초과됩니다. 데이터 저장은 반드시 정해진 파이프라인(예: `syncToDatabase` 함수)을 통해서만 한 줄로 통과해야 합니다.
